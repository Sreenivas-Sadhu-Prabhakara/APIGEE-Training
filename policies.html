Policy explanation and recipes

Apigee policy cookbook

Each recipe includes the teaching point, when to use it, and starter XML. Labs combine these into reusable shared flows and security tiers.

Verify API key

Use for low-risk APIs, internal prototypes, portal onboarding, or as an additional app identity signal.

<VerifyAPIKey name="VAK-Verify-Client">
  <APIKey ref="request.header.x-api-key"/>
</VerifyAPIKey>

OAuth access token

Use where Apigee issues or validates Apigee-managed OAuth tokens and you need scopes, product, app, developer, and quota context.

<OAuthV2 name="OA-Verify-Access-Token">
  <Operation>VerifyAccessToken</Operation>
  <AccessTokenPrefix>Bearer</AccessTokenPrefix>
  <Scope>accounts:read</Scope>
</OAuthV2>

Verify external JWT

Use for tokens issued by an external identity provider. Keep issuer, audience, allowed algorithms, and JWKS location explicit.

<VerifyJWT name="JWT-Verify-Partner-Token">
  <Algorithm>RS256</Algorithm>
  <Source>request.header.authorization</Source>
  <PublicKey>
    <JWKS uri="https://idp.example.com/.well-known/jwks.json"/>
  </PublicKey>
  <Issuer>https://idp.example.com</Issuer>
  <Audience>api://banking-accounts</Audience>
  <Subject ref="jwt.JWT-Verify-Partner-Token.claim.sub"/>
</VerifyJWT>

Spike arrest

Use for smoothing bursts. Do not use it as the only source of accurate business entitlement counting.

<SpikeArrest name="SA-Burst-Protection">
  <Identifier ref="client_id"/>
  <Rate>30ps</Rate>
  <UseEffectiveCount>true</UseEffectiveCount>
</SpikeArrest>

Quota

Use for product entitlements, partner contracts, and API plan enforcement.

<Quota name="Q-Product-Daily-Limit">
  <Identifier ref="client_id"/>
  <Allow count="10000"/>
  <Interval>1</Interval>
  <TimeUnit>day</TimeUnit>
  <Distributed>true</Distributed>
  <Synchronous>true</Synchronous>
</Quota>

JSON threat protection

Use to constrain structural abuse before the payload reaches application code.

<JSONThreatProtection name="JTP-Request-Shape">
  <ArrayElementCount>100</ArrayElementCount>
  <ContainerDepth>8</ContainerDepth>
  <ObjectEntryCount>60</ObjectEntryCount>
  <ObjectEntryNameLength>64</ObjectEntryNameLength>
  <StringValueLength>4096</StringValueLength>
</JSONThreatProtection>

Assign message

Use for clean request shaping: correlation IDs, canonical headers, and backend-specific details hidden from clients.

<AssignMessage name="AM-Set-Request-Headers">
  <AssignTo createNew="false" transport="http" type="request"/>
  <Set>
    <Headers>
      <Header name="x-correlation-id">{request.header.x-correlation-id}</Header>
      <Header name="x-apigee-proxy">banking-accounts-v1</Header>
    </Headers>
  </Set>
  <IgnoreUnresolvedVariables>true</IgnoreUnresolvedVariables>
</AssignMessage>

Raise fault

Use for standardized fail-closed responses and for removing accidental backend leakage.

<RaiseFault name="RF-Invalid-Consent">
  <FaultResponse>
    <Set>
      <StatusCode>403</StatusCode>
      <ReasonPhrase>Forbidden</ReasonPhrase>
      <Payload contentType="application/json">
        {"error":"consent_required","message":"Valid consent context is required."}
      </Payload>
    </Set>
  </FaultResponse>
</RaiseFault>

Policy choreography rule of thumb

  1. Reject structurally unsafe traffic before expensive work.
  2. Authenticate the caller before using identity-dependent counters.
  3. Authorize scopes, resources, and consent before routing.
  4. Normalize headers and payloads only after the request is trusted enough to process.
  5. Log decisions, not secrets.
