Architecture and protocol grounding

Why Apigee architecture works

This page gives learners the mental model underneath the labs: three-tier API architecture, REST as an architectural style, and the RFCs that define the Web protocol surface most Apigee proxies manage.

Three-tier architecture

Separate experience, policy, and implementation.

Apigee belongs in the API management tier. It should protect and mediate the contract, but it should not become the system of record, authorization server, consent database, or business transaction engine.

1 Experience tier

Mobile, web, partner, B2B, internal tools, portal apps, and API consumers.

2 API management tier

Apigee X proxies, products, policies, shared flows, analytics, threat protection, mediation, and routing.

3 Service and data tier

Domain services, legacy systems, Cloud Run, GKE, databases, consent store, identity provider, logs, and monitoring.

REST and RFC clarity

Teach REST precisely.

REST is not itself an RFC. REST is Roy Fielding's architectural style for network-based software architectures. The protocols and identifiers used by RESTful HTTP APIs are standardized in RFCs such as HTTP Semantics and URI Generic Syntax.

Fielding dissertation

Architectural Styles and the Design of Network-based Software Architectures introduced REST as constraints for scalable, independently deployable, intermediary-friendly distributed hypermedia systems.

RFC 9110: HTTP Semantics

Defines the architecture, terminology, methods, status codes, headers, representations, caching hooks, and the http/https URI schemes shared across HTTP versions.

RFC 3986: URI Generic Syntax

Defines the generic syntax for identifiers, including scheme, authority, path, query, fragment, percent encoding, hierarchy, and security considerations.

MethodHTTP method semantics

Use RFC 9110 to teach GET safety, PUT idempotency, POST processing, DELETE semantics, and status code discipline.

RepresentationJSON and media types

Teach contract stability, content negotiation, schema validation, and response shaping through Apigee mediation.

IntermediaryApigee policies

Teach that Apigee adds security, routing, analytics, transformation, and caching while preserving client/backend independence.

RESTful resource checklist

# Resource design review
- Resource names are nouns, not actions.
- Paths identify stable resources.
- HTTP methods carry action semantics.
- GET does not mutate server state.
- PUT and DELETE behavior is idempotent where used.
- POST is used for processing or subordinate resource creation.
- Status codes match the outcome.
- Error payloads are consistent.
- Sensitive values are not placed in URI query strings.
- Cache behavior is explicit.

Apigee placement checklist

# What belongs in Apigee
- Authentication and token validation
- Product, app, scope, and quota enforcement
- Burst protection and threat protection
- Header and payload mediation
- Routing and target abstraction
- Correlation IDs, analytics, logging hooks

# What usually belongs elsewhere
- User login and consent capture
- Core business authorization decisions
- Ledger updates and system-of-record writes
- Fraud scoring models
- Long-running orchestration state
